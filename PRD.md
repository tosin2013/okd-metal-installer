# Product Requirements Document: OKD-Metal

**Status:** Scoping
**Target Release:** v0.1.0 (MVP)

## 1. Introduction & Overview

OKD-Metal (repository name `okd-metal-installer`) is envisioned as a lightweight, self-hosted, and fully automated bare-metal provisioning tool specifically designed for OKD (the community distribution of Red Hat OpenShift). It aims to provide a community-friendly, disconnected-capable alternative to the Red Hat SaaS Assisted Installer, filling a critical gap in the current OKD ecosystem. The primary goal of OKD-Metal is to simplify and automate the deployment of OKD on bare-metal infrastructure, making it more accessible to a wider range of users and environments.

## 3. Problem Statement

The current process for deploying OKD on bare-metal infrastructure is fraught with challenges that hinder its adoption and usability. The official Red Hat Assisted Installer, while powerful, is tightly integrated with commercial OpenShift SaaS endpoints, requiring enterprise pull secrets and complex workarounds to function with OKD. This creates a significant barrier for community users and those in disconnected environments. Furthermore, recent regressions in the FCOS/SCOS operating system images have introduced instability into the bootstrap pivoting process, making installations unreliable [1]. The community urgently needs a dedicated, turnkey solution that streamlines bare-metal OKD installations without the overhead and dependencies of enterprise-focused frameworks.

## 4. Use Cases / User Stories

*   **As an Edge Infrastructure Engineer,** I want to declaratively define my bare-metal hosts in a single configuration file so that I can automate the provisioning of a new OKD cluster with minimal manual intervention.
*   **As a Homelabber,** I want to generate a single bootable ISO that automatically discovers my hardware, fetches the correct configuration, and installs OKD without requiring deep technical knowledge of the underlying processes.
*   **As a Disconnected Environment Admin,** I want to provision an OKD cluster in a fully air-gapped environment so that I can maintain security and compliance without relying on internet connectivity.
*   **As an OKD user,** I want to deploy a Single-Node OKD (SNO) cluster on a single physical server to minimize my hardware footprint and operational costs for smaller-scale deployments.
*   **As a developer working on OKD-Metal,** I want to be able to get VNC access to my development machine hosted on Hetzner, so that I can debug low-level OS and boot-time issues during the development of the custom discovery agent.

## 6. Technical Architecture & Constraints

OKD-Metal will be built upon a foundation of established, community-trusted tools. The architecture is designed to be lightweight and modular, avoiding complex dependencies. Key technical constraints and components include:

*   **`coreos-installer`:** This will be used to customize the FCOS live ISO, embedding the discovery agent and necessary networking configurations.
*   **Ignition:** The tool will generate and serve Ignition configurations, which are used to provision the OKD nodes. The system must handle potential version incompatibilities between Ignition specifications.
*   **FCOS/SCOS Compatibility:** The solution must be robust to changes and potential regressions in the underlying Fedora CoreOS (FCOS) and the new CentOS Stream-based (SCOS) operating systems, particularly concerning the bootstrap pivot process [1].
*   **`openshift-install`:** OKD-Metal will wrap the `openshift-install` binary, orchestrating its execution to generate the necessary cluster configurations.
*   **NMState:** The Kubernetes NMState operator will be leveraged for declarative network configuration, allowing for the pre-configuration of complex networking setups.
*   **Jumpbox Architecture:** A separate, dedicated jumpbox server will be provisioned for development and management access. This jumpbox will have VNC enabled, providing a graphical environment for interacting with the cluster and debugging. This approach isolates the development environment from the cluster itself, improving security and stability.
*   **Route 53 Integration:** For DNS, the tool will integrate with AWS Route 53. It will automatically create the required `A` and `PTR` records for the cluster nodes and API endpoints. This requires the user to provide AWS credentials with the necessary permissions.
*   **Cluster Topologies:** The installer will be designed to be flexible and support various cluster sizes. The default will be a Single-Node OKD (SNO) deployment. Additionally, it will support a 3-node compact cluster (where control plane and worker roles are co-located) and a full high-availability (HA) cluster with dedicated control plane and worker nodes.

## 7. Out of Scope (MVP)

The initial MVP release of OKD-Metal will focus on the core provisioning workflow. The following features will be considered out of scope for the first version:

*   **Day-2 Operations:** Post-installation cluster management, such as node scaling, upgrades, and monitoring.
*   **Cloud Provider Integrations:** Support for deploying OKD on public or private cloud platforms.
*   **Advanced UI/Dashboard:** A sophisticated graphical user interface for managing the provisioning process.
*   **Multi-cluster Management:** The ability to manage and provision multiple OKD clusters from a single control plane.

## 8. Success Metrics

The success of OKD-Metal will be measured by its ability to simplify and improve the bare-metal installation experience for the OKD community. Key metrics will include:

*   **Time to Deploy:** A significant reduction in the time and effort required to deploy a bare-metal OKD cluster compared to existing manual methods.
*   **Reduction in Deployment Failures:** A measurable decrease in the rate of installation failures, particularly those related to bootstrap and networking issues.
*   **Community Adoption:** The number of active users, contributors, and community-created documentation and tutorials.
*   **User Satisfaction:** Positive feedback from the community, gathered through surveys, GitHub issues, and forum discussions.

# 10. Ignition Configuration Generation and Management

**Status**: Accepted
**Date**: 2026-03-16
**Domain**: Configuration Management, OS Provisioning

## Context

OKD nodes are provisioned using Ignition configurations -- JSON documents that specify the complete desired state of a CoreOS system at first boot (users, files, systemd units, storage layout). These configs are generated by the `openshift-install` binary from an `install-config.yaml` and must be served to nodes during the discovery/installation process.

The Ignition configs contain sensitive material (certificates, kubeconfig, bootstrap tokens) and are specific to each node role (bootstrap, master, worker). They also have a built-in expiry (24 hours by default for the certificates), creating a time-sensitive deployment window.

## Decision

Wrap the `openshift-install` binary in an Ansible role (`ignition_generate`) that:

1. **Templates `install-config.yaml`** from Ansible variables (cluster name, base domain, pull secret, SSH key, network config, node counts, topology)
2. **Invokes `openshift-install create ignition-configs`** (or `create single-node-ignition-config` for SNO) to produce the Ignition files
3. **Stages Ignition configs** to a directory served by a lightweight HTTP server on the jumpbox/provisioning host
4. **Manages config lifecycle**: detects expired configs, supports regeneration, archives old configs
5. **Handles Ignition spec version differences** between FCOS and SCOS variants (per ADR-008)

The HTTP serving component will be a simple nginx or Python HTTP server managed by Ansible, listening on a configurable port. The discovery agent (ADR-002) fetches the appropriate Ignition config by role from this server.

## Consequences

### Positive

- Full automation of the Ignition generation workflow; no manual `openshift-install` invocation
- Templated `install-config.yaml` ensures consistency with the rest of the Ansible-managed configuration
- HTTP serving of Ignition configs is simpler and more debuggable than embedding them in the ISO
- Config lifecycle management prevents silent failures from expired certificates
- Role-based config selection (master/worker/bootstrap) is handled automatically

### Negative

- `openshift-install` is a large, opaque binary; its behavior and flags change across releases
- 24-hour certificate expiry creates urgency in the deployment workflow
- HTTP server on the jumpbox must be reachable from all booting nodes (network topology constraint)
- Ignition configs contain secrets; the HTTP server must be appropriately secured (TLS, network isolation, or time-limited availability)

## Implementation Plan

1. Create `roles/ignition_generate/` Ansible role
2. Template `install-config.yaml.j2` with all required variables
3. Add tasks to invoke `openshift-install` and capture output
4. Create `roles/config_serve/` role for HTTP server setup (nginx)
5. Add config expiry detection and regeneration logic
6. Implement config staging with proper file permissions (0600 for secrets)
7. Add integration with discovery agent phone-home URL construction

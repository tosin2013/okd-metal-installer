# 2. Custom Discovery Agent and Boot Media

**Status**: Accepted
**Date**: 2026-03-16
**Domain**: Boot Media, OS Provisioning

## Context

OKD-Metal needs a mechanism to boot bare-metal hosts, discover their hardware, fetch the appropriate Ignition configuration, flash the disk, and reboot into the installed OS. The industry-standard approach for Fedora CoreOS (FCOS) provisioning uses `coreos-installer` to customize live ISOs with embedded scripts and Ignition configs.

The Red Hat Assisted Installer solves this with a SaaS-backed discovery agent, but that approach requires internet connectivity and enterprise pull secrets -- both of which conflict with OKD-Metal's disconnected-capable, community-friendly goals.

## Decision

Build a custom discovery agent embedded into a FCOS live ISO using `coreos-installer iso customize`. The agent will be a shell script (or minimal binary) that runs during the initramfs phase and performs:

1. Hardware telemetry collection (MAC addresses, disk inventory, memory, CPU)
2. Phone-home to the provisioning host to register and retrieve the correct Ignition payload
3. Disk flashing via `coreos-installer install` with the retrieved Ignition config
4. Reboot into the installed system

The ISO will be generated by an Ansible role (`iso_customize`) that:
- Downloads the appropriate FCOS/SCOS live ISO
- Embeds the discovery agent script
- Embeds pre-compiled NetworkManager `.nmconnection` files for network bootstrapping
- Embeds any CA certificates required for disconnected environments

## Consequences

### Positive

- Full control over the discovery and provisioning flow without SaaS dependencies
- Single bootable ISO works for all node roles (control plane, worker, SNO)
- Supports disconnected environments by embedding all required artifacts
- `coreos-installer iso customize` is an upstream-supported, stable API
- Discovery agent can be iterated independently of the rest of the stack

### Negative

- Must maintain a custom discovery agent script across FCOS/SCOS releases
- ISO must be regenerated when cluster configuration changes
- Phone-home mechanism requires a minimal HTTP server running on the provisioning host (can be a simple `python -m http.server` or Ansible-managed nginx)
- Agent debugging requires VNC or serial console access to the booting host

## Implementation Plan

1. Create `roles/iso_customize/` Ansible role
2. Implement discovery agent script at `files/discovery-agent.sh`
3. Template the phone-home URL and cluster-specific parameters
4. Add tasks to download FCOS ISO, embed agent, and produce final ISO
5. Integrate with the `network_configure` role for nmconnection embedding
